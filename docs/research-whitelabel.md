# Отчёт исследования: whitelabel

Исследование лучших практик white-label архитектуры (2024–2026) показывает консенсус: продукт для перепродажи должен строиться по принципу «конфигурация как данные, а не код» (configuration as data, not code). Весь брендинг выносится в трёхслойную систему design tokens на CSS custom properties: примитивы (--color-blue-500: #2980b9) → семантические токены (--color-primary: var(--color-blue-500)) → компонентные токены (--button-background: var(--color-interactive)). При такой структуре смена бренда сводится к переопределению примитивов в одном месте — каскад CSS автоматически пересчитывает все стили без перекомпиляции. Токены хранятся в одном JSON/YAML-файле (стандарт W3C DTCG, инструмент Style Dictionary генерирует из него CSS/iOS/Android-выходы), а brand config дополняется логотипом, фавиконом, названием продукта, шрифтами, доменом, email-шаблонами и реквизитами. Codemagic и практика React/Webpack подтверждают паттерн: один базовый код + файл конфигурации клиента (assets, цвета, имя, API-endpoints), применяемый скриптом сборки или в рантайме.

По архитектуре для продаваемого продукта однозначно выигрывает модульный монолит: по данным CNCF-опроса 2025 года 42% организаций консолидируют микросервисы обратно, инфраструктура микросервисов стоит в 3,75–6 раз дороже (≈$15K/мес против $40–65K/мес на сопоставимой функциональности), а монолит разворачивается одной командой (docker-compose up) без Kubernetes, service mesh и платформенной команды — это критично для покупателя с ограниченными DevOps-ресурсами. Границы модулей оформляются как явные внутренние API (пример Shopify: домены Shop/Catalog/Checkout/Payments с инструментом контроля границ Packwerk). Модули включаются/выключаются feature-флагами в конфигурационных файлах — это же рекомендует и российская практика on-premise поставок (Хабр): флаги хранятся в конфиге/БД, управляются централизованно, публичные параметры отделены от внутренних. Наружу продукт отдаёт версионированный REST API с OpenAPI-спецификацией (актуальна 3.1/3.2), генерируемой из кода в CI — это машиночитаемый контракт и стандарт де-факто передачи интеграционной документации. Мультиарендность «с первого дня» не обязательна для продукта, продаваемого одной компании: достаточно дизайна «single-tenant с tenant_id в схеме БД» (изоляция через tenant_id/row-level security закладывается в модель данных, но инфраструктурно продукт ставится как одиночная инсталляция) — это и проще для покупателя, и оставляет путь к multi-tenant.

Юридически по российскому праву есть два пути: договор об отчуждении исключительного права (ст. 1234 ГК РФ — покупатель получает права полностью, продавец теряет возможность перепродажи; для коммерческих структур только возмездный; если ПО регистрировалось в Роспатенте, переход регистрируется) и лицензионный договор (исключительная/неисключительная лицензия — продавец сохраняет права и может продавать другим). Передачу исходного кода фиксируют отдельным актом приёма-передачи с описью; обязательно раскрытие лицензий third-party зависимостей (open-source аудит). Пакет передачи по чек-листам 2025–2026: исходный код (совпадающий с продакшен-версией), репозиторий с историей, архитектурный документ, описание стека и модулей, схема БД + миграции, OpenAPI-документация, deployment-инструкции (проверенные «незнакомым человеком»), env-конфигурация по окружениям, тестовая документация, учётки/доступы, описание безопасности (аутентификация, шифрование, аудит-логи) и сессии передачи знаний с SLA на период поддержки.

Специфика российского покупателя (2025–2026): продукт должен ставиться on-premise самодостаточным дистрибутивом (Docker-образы со всеми зависимостями внутри, работа в air-gapped контуре без доступа к интернету), на стеке без санкционных рисков — PostgreSQL как БД де-факто импортозамещения (поддерживается Postgres Pro/Arenadata из реестра отечественного ПО), без жёстких зависимостей от Firebase/AWS/Stripe/Google Fonts (шрифты и ассеты — локально). Соответствие 152-ФЗ обеспечивается возможностью размещения данных в РФ. Типовые требования корпоративных заказчиков: LDAP/Active Directory, SSO (SAML/OAuth2/Kerberos), syslog-мониторинг, резервное копирование, аудит-лог входов и изменений прав, ручные миграции БД недопустимы — всё автоматизировано. Идеал «тиражируемости»: развёртывание у типового заказчика без доработок, а смена брендинга — правка одного brand-config файла и подмена папки с ассетами.

## Факты
- ЧЕК-ЛИСТ — Брендинг: весь визуал построен на трёхслойных design tokens через CSS custom properties (примитивы → семантические → компонентные токены); смена бренда = переопределение примитивов в одном месте, каскад CSS пересчитывает всё без перекомпиляции и правок компонентов.
- ЧЕК-ЛИСТ — Единый brand config: один файл (JSON/YAML, желательно в формате W3C DTCG) содержит цвета, шрифты, логотип, фавикон, название продукта, домен, email-шаблоны, контакты; null-значение поля = наследование дефолта; Style Dictionary генерирует из токенов CSS-переменные и нативные темы для iOS/Android.
- ЧЕК-ЛИСТ — Название продукта и логотип нигде не захардкожены: продуктовое имя — переменная (APP_NAME/brand.productName), используемая в UI, title, письмах, PDF-отчётах и системных уведомлениях; ассеты лежат в отдельной brand-папке, заменяемой целиком.
- ЧЕК-ЛИСТ — Принцип «configuration as data, not code»: всё клиентское поведение (брендинг, фичи, интеграции, лимиты) меняется конфигурацией без деплоя; категорически избегать форков кодовой базы под клиента — это главная названная в источниках ошибка white-label.
- ЧЕК-ЛИСТ — Архитектура: модульный монолит, а не микросервисы — по CNCF-опросу 2025 года 42% организаций консолидируют микросервисы обратно; инфраструктура микросервисов в 3,75–6 раз дороже (≈$15K/мес против $40–65K/мес), а монолит покупатель разворачивает одной командой docker-compose без Kubernetes и платформенной команды.
- ЧЕК-ЛИСТ — Границы модулей: домены (auth, каталог, платежи, отчёты и т.д.) изолированы с явными внутренними API (паттерн Shopify/Packwerk); каждый модуль описан в документации: назначение, зависимости, точки расширения.
- ЧЕК-ЛИСТ — Feature flags: каждый функциональный модуль включается/выключается флагом в конфигурационном файле или таблице БД (per-tenant entitlements); флаги управляются централизованно, изменение тарифа/комплектации = правка конфига, не кода; избегать «разрастания флагов» (flag sprawl) — флаги документируются и чистятся.
- ЧЕК-ЛИСТ — API: версионированный REST API с автогенерируемой из кода OpenAPI 3.1+ спецификацией (single source of truth, контракт для интеграции); Swagger UI/Redoc в комплекте; обратная совместимость между минорными версиями; вебхуки/события для расширения покупателем без правки ядра.
- ЧЕК-ЛИСТ — Мультиарендность не обязательна с первого дня для продукта, продаваемого одной компании: достаточно заложить tenant_id в схему БД и middleware-фильтрацию (опционально row-level security в PostgreSQL) — инсталляция работает как single-tenant, но путь к multi-tenant открыт без переписывания модели данных.
- ЧЕК-ЛИСТ — Лицензирование по российскому праву: либо договор об отчуждении исключительного права (ст. 1234 ГК РФ — полная передача прав, для коммерческих организаций только возмездный, при регистрации ПО в Роспатенте переход права регистрируется), либо лицензионный договор (исключительная лицензия — покупатель единственный пользователь, неисключительная — можно продавать нескольким компаниям).
- ЧЕК-ЛИСТ — Передача кода оформляется отдельным актом приёма-передачи с описью носителей и репозиториев; обязателен аудит и раскрытие лицензий всех open-source зависимостей (нет GPL-заражения, нет компонентов с запретом коммерческого использования), плюс подтверждение цепочки авторских прав (договоры с разработчиками).
- ЧЕК-ЛИСТ — Пакет передачи (handover package): исходный код всех частей, совпадающий с продакшен-версией; git-репозиторий с историей и описанием веток; архитектурный документ и описание технологического стека; схема БД + скрипты миграций + бэкап с тестовыми данными; OpenAPI-документация; README с установкой dev-окружения; деплой-инструкции; конфигурация окружений (dev/stage/prod) и список всех env-переменных с описанием.
- ЧЕК-ЛИСТ — В пакет также входят: тестовая документация (тест-кейсы, автотесты, результаты нагрузочного и security-тестирования), описание механизмов безопасности (аутентификация, шифрование, аудит-логирование), CI/CD-конфигурация, учётные данные и передача владения аккаунтами (хостинг, домены, SSL), сессии передачи знаний и SLA на гарантийный период поддержки.
- ЧЕК-ЛИСТ — Деплой-инструкция должна быть проверена «холодным» человеком: лучшая практика — дать развернуть продукт сотруднику, не знакомому с проектом, строго по документации; целевой показатель — установка с нуля за часы, а не дни.
- ЧЕК-ЛИСТ — Поставка для российской компании: самодостаточный on-premise дистрибутив (Docker-образы/docker-compose со всеми зависимостями внутри), работа в изолированном контуре без интернета (air-gapped): никаких CDN, Google Fonts, внешних API в рантайме — все шрифты, библиотеки и ассеты в комплекте.
- ЧЕК-ЛИСТ — Стек без санкционных рисков: PostgreSQL как СУБД (стандарт импортозамещения в РФ, совместима с Postgres Pro из реестра отечественного ПО), открытые технологии (Linux, nginx, Redis), отсутствие жёстких зависимостей от AWS/Firebase/Stripe/западных SaaS; интеграции с внешними сервисами — через заменяемые адаптеры (паттерн «порты и адаптеры»).
- ЧЕК-ЛИСТ — Соответствие 152-ФЗ: все персональные данные хранятся в БД инсталляции покупателя (на серверах в РФ), нет скрытой телеметрии и отправки данных наружу; документировано, какие ПДн обрабатываются и где лежат.
- ЧЕК-ЛИСТ — Типовые корпоративные интеграции российского заказчика: LDAP/Active Directory для каталога пользователей, SSO по SAML/OAuth2/Kerberos (использовать готовые решения типа Keycloak, не самописную аутентификацию), SMTP-настройка для своего почтового сервера, syslog для мониторинга, штатное резервное копирование, аудит-лог входов/выходов и изменения прав.
- ЧЕК-ЛИСТ — Эксплуатация: автоматические миграции БД при обновлении (ручные недопустимы), конфигурируемые уровни логирования без утечки чувствительных данных, сайзинг-документ с расчётом аппаратных требований; публичные конфиг-параметры отделены от внутренних, чтобы заказчик их случайно не сломал.
- ЧЕК-ЛИСТ — Критерий готовности (тиражируемость): новый покупатель получает свой бренд правкой одного brand-config (цвета, логотип, шрифты, название) + одного env-файла (домен, БД, SMTP, ключи) + заменой папки ассетов; всё остальное — включение нужных модулей feature-флагами; ни одной правки исходного кода для типового внедрения.
- Факт: иерархия кастомизации по уровню риска (Developex 2026): 1) визуал/брендинг через CSS-переменные и метаданные (низкий риск), 2) включение функций feature-флагами (средний), 3) конфигурируемые workflow/роли (средний), 4) глубокие интеграции через плагины и версионированные API (высокий, требует governance).
- Факт: семь типовых ошибок white-label по источникам — ловушка «безграничной кастомизации», ручная настройка клиентов вместо автоматизации, хардкод-форки под клиентов, недостаточная изоляция данных, безопасность «потом», разрастание feature-флагов, игнорирование релиз-менеджмента и обратной совместимости.
- Факт: Amazon Prime Video (команда Video Quality Analysis) сократила инфраструктурные расходы на 90%, мигрировав с микросервисов обратно на монолит — аргумент в пользу простой архитектуры для передаваемого продукта.
- Факт: актуальная версия OpenAPI — 3.2.0 (сентябрь 2025); лучшая практика — генерация спецификации из аннотаций кода в CI, чтобы документация никогда не расходилась с реализацией; детальные описания операций также делают API пригодным для AI-агентов.

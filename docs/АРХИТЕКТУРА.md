# Архитектура

## Стек и мотивация выбора

| Слой | Технология | Почему |
| --- | --- | --- |
| Сервер | Node.js 22+, Express 5, TypeScript | самый распространённый стек у российских веб-команд — покупателю легко найти разработчиков |
| БД | `node:sqlite` (встроенный) | ноль нативных зависимостей: `npm install` без компиляторов; для продакшена — миграция на PostgreSQL (требование реестра отечественного ПО), см. ИНТЕГРАЦИЯ.md |
| Аутентификация | scrypt (`node:crypto`) + HMAC-cookie | без внешних библиотек: меньше поверхность атаки и проще аудит ИБ |
| Фронтенд | React 18, Vite, TypeScript | стандарт отрасли; лёгкий бандл, целевая работа на слабых машинах |
| Стили | чистый CSS + дизайн-токены | без UI-фреймворков: полный контроль брендирования через CSS-переменные |

Внешних рантайм-зависимостей сервера всего две: `express` и `multer`.
Это осознанное решение под продажу: SBOM минимален, GPL-заражения нет,
аудит зависимостей тривиален.

## Модульный монолит

Сервер собирается из модулей-роутеров в `server/src/index.ts`. Каждый модуль:

- лежит в своей папке `server/src/modules/<имя>/`;
- экспортирует Express Router;
- работает со своей зоной таблиц БД;
- проверяет права через общий `core/access.ts`.

Удаление модуля = удаление одной строки `app.use(...)` и папки.
Добавление = новая папка + одна строка. Фичефлаги в `brand.config.json`
отключают модули без правки кода (проверки на сервере И в интерфейсе).

```
index.ts ──┬── auth ──────────── сессии, регистрация, профиль
           ├── courses ───────── курсы, участники, коды, копирование
           ├── topics ────────── темы-разделы заданий
           ├── coursework ────── задания/тесты/вопросы/материалы, рубрики
           ├── quizzes ───────── вопросы тестов, автопроверка
           ├── submissions ───── сдачи, оценки, возврат, события
           ├── announcements ─── лента объявлений
           ├── comments ──────── публичные и приватные комментарии
           ├── grades ────────── журнал оценок
           ├── files ─────────── загрузка/скачивание с контролем прав
           ├── notifications ─── уведомления + каналы доставки
           ├── planner ───────── список дел, календарь
           └── comment-bank ──── банк комментариев учителя
core/ ─────── db, schema.sql, access, errors, validate, scheduler
```

## Схема данных

Модель повторяет проверенные паттерны Google Classroom API:

```
users ──< course_members >── courses ──< topics
                                │
                                ├──< announcements ──< comments(ANNOUNCEMENT)
                                │
                                └──< coursework ──< quiz_questions
                                       │  ├──< coursework_assignees (адресация)
                                       │  ├──── rubrics ──< criteria ──< levels
                                       │  └──< comments(COURSEWORK)
                                       │
                                       └──< submissions ──< quiz_answers
                                              ├──< submission_events (история)
                                              ├──< submission_rubric_grades
                                              └──< comments(SUBMISSION — приватные)
attachments — полиморфные вложения (owner_type + owner_id)
notifications, comment_bank — пользовательские данные
```

Ключевые паттерны:

1. **Роль — атрибут связи, не пользователя.** `course_members.role`:
   один человек — учитель в одном курсе и ученик в другом.
2. **Двухфазные оценки.** `draft_grade` (виден учителю) → `grade`
   (публикуется «возвратом» работы). Как в Classroom.
3. **Append-only история.** `submission_events` фиксирует сдачи, возвраты
   и изменения оценок с указанием действующего лица.
4. **Полиморфные вложения.** Один механизм для заданий, объявлений и сдач;
   файл «закрепляется» за владельцем после загрузки (см. МОДУЛИ.md → files).
5. **Состояния постов.** DRAFT → SCHEDULED → PUBLISHED; планировщик
   (`core/scheduler.ts`) публикует отложенное и рассылает уведомления.
6. **Хранение оценок всегда в баллах.** Шкала курса (5-балльная/проценты) —
   только отображение (`web/src/utils.ts → formatGrade`).

## Поток запроса

```
HTTP → express.json → attachUser (cookie → req.user)
     → Router модуля → requireAuth → requireMember/requireTeacher (права)
     → валидация (core/validate) → SQL (core/db) → JSON-ответ
     → errorHandler (единый формат ошибок { error: { code, message } })
```

Express 5 пробрасывает асинхронные исключения в errorHandler автоматически —
обёртки try/catch в маршрутах не нужны.

## Безопасность

- Пароли: scrypt с солью (`auth/passwords.ts`), сравнение через `timingSafeEqual`.
- Сессии: httpOnly + SameSite=Lax cookie, HMAC-SHA256-подпись, TTL 30 дней;
  секрет — из `SESSION_SECRET` или автогенерируется в `server/data/`.
- Каждый маршрут проверяет членство в курсе и роль; ученик не видит чужие
  сдачи, черновики, приватные комментарии и правильные ответы тестов.
- Файлы недоступны без проверки прав на «владельца» вложения.
- Никакой телеметрии наружу; все данные — в инсталляции владельца (152-ФЗ).

## Производительность

- Bundle-бюджет: без тяжёлых UI-библиотек, системные шрифты, иконки — SVG-инлайн.
- SQLite в режиме WAL; индексы под основные выборки (см. schema.sql).
- Целевой профиль: школа на слабом ПК и медленном канале — главный вывод
  исследования болей ру-сегмента (см. research-ru-needs.md).
